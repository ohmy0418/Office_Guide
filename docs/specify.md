# 회사생활가이드 v1 통합 명세서

## 1. 문서 목적
이 문서는 설계안, PRD, FRD, TRD를 기준으로 v1 개발에 바로 사용할 수 있는 통합 명세를 정리한 문서다.
제품 목표, 범위, 기능 기준선, 기술 기준선, 데이터 구조, API, 운영 기준을 하나의 문서로 합쳤다.
문서 간 불일치나 미확정 항목은 `[확인 필요]` 프리픽스로 표시한다.

---

## 2. 제품 개요
회사생활가이드는 사내 비정형 문서와 정형 데이터를 함께 탐색하여, 자연어 질문에 근거 기반으로 답변하는 검색 중심 RAG 시스템이다.
시스템은 다음 흐름으로 동작한다.

#### 메인 처리 흐름
`질문 접수 → 핵심 정보 추출 → 라우팅 → 문서 검색 또는 정형 조회 또는 혼합 조회 → 결과 병합 → 답변 생성 → 응답 반환`

#### 예외 처리 흐름
`검색 결과 부족 / 라우팅 실패 / 조회 실패 / 권한 없음 발생 시 → fallback 또는 예외 응답 생성 → 로그 저장`

### 2.1 제품 목표
1. 사내 규정, 절차, 가이드 문서의 검색 정확도를 높인다.
2. 사용자가 자연어 질문만으로 필요한 문서 근거와 담당 정보를 빠르게 찾을 수 있게 한다.
3. 문서형 질문, 정형 조회형 질문, 혼합 조회형 질문을 하나의 시스템 안에서 처리한다.
4. 최신 문서 우선 반영과 권한 기반 접근 제어를 지원한다.
5. 답변 가능성이 낮을 경우 fallback으로 안전하게 응답한다.

### 2.2 기본 원칙
- 검색 중심 RAG 원칙
- 근거 우선 원칙
- 권한 우선 원칙
- 안전 응답 원칙
- 초기 단순성 원칙

---

## 3. 범위

### 3.1 포함 범위
- 비정형 문서 업로드, 저장, 변환, OCR, 정제, chunk 분할, 임베딩, 검색용 인덱싱
- 사용자 질문 분석 및 라우팅
- 문서 검색(RAG)
- 정형 데이터 조회(DB/API)
- 혼합 조회 결과 병합
- 근거 기반 답변 생성
- fallback 및 예외 처리
- 관리자 운영 기능(업로드·반영 상태 확인 등, 아래 4.2)
- 운영 로그 및 품질 평가 로그

### 3.2 제외 범위
- AI의 자유로운 자동 SQL 생성 및 실행
- PDF 일부 수정사항만 반영하는 증분 색인
- 검색 랭킹 가중치 자동 튜닝
- 자유로운 범용 검색 엔진
- 근거 없는 오픈형 챗봇 응답
- **동의어/사전의 운영 관리 기능**(UI·API·운영 주체 프로세스) — v1 제외. 질문 정규화용 **코드 내 고정 치환**만 허용

---

## 4. 사용자와 권한

### 4.1 일반 사용자
- 규정, 절차, 가이드, 담당 부서/담당자, 조직/구성원 정보를 조회할 수 있다.
- v1에서는 공개 대상 문서 및 정형 데이터를 조회할 수 있다.

### 4.2 관리자
- **문서 업로드**
- **반영·처리 상태 확인**(진행 단계, 완료/실패, **실패 사유·오류 코드** 포함)

인제스트(반영 실행) API는 관리자 전용으로 두되, v1 관리자 업무 범위는 위 두 가지 축(업로드·상태 확인)으로 한정한다.

### 4.3 권한 원칙
- v1에서는 콘텐츠 ACL을 적용하지 않으며, **운영 기능에만 관리자 권한**을 적용한다.
- 관리자 전용 기능은 일반 사용자에게 노출하지 않는다.
- 권한 검증은 운영 기능(API, 운영 화면)에 적용한다.

## 5. 질문 유형

### 5.1 문서형 질문
규정, 절차, 정책, 가이드 등 문서 근거가 필요한 질문
- 예: 기숙사 신청 절차 알려줘
- 예: 장비 교체 기준이 뭐야

### 5.2 정형 조회형 질문
임직원, 조직, 연락처, 소속 등 구조화된 데이터 조회 질문
- 예: 홍길동 이메일 알려줘
- 예: 피플팀 담당자 누구야

### 5.3 혼합 조회형 질문
문서 근거와 정형 데이터가 함께 필요한 질문
- 예: 장비 구매 비용 기준과 담당자 정보를 알려줘
- 예: 기숙사 신청 기준과 담당 부서 알려줘

### 5.4 처리 불가/범위 외 질문
서비스 범위 밖 질문, 라우팅 실패 질문, 충분한 근거가 확보되지 않은 질문

## 6. v1 기준선

### 6.1 확정 기준
- 정형 조회는 사전 정의 SQL 또는 API 중심으로 시작한다.
- 문서 검색은 하이브리드 검색(키워드 검색 + 유사도 검색)을 기본으로 한다.
- 혼합 조회는 기본적으로 병렬 처리하고, 의존 관계가 명확한 경우에만 순차 처리한다.
- PDF 수정 반영은 전체 재색인을 기본으로 한다.
- 운영 로그와 품질 평가 로그는 분리 저장한다.

### 6.2 검색 기준선
- 문서 후보 수: top 3
- chunk 후보 수: top 10
- rerank 대상 수: 6
- 최종 context 기본값: 상위 4개 chunk
- 품질 지표 목표: Recall@k 우선, Recall 최소 0.8 지향

### 6.3 결합 전략
- v1 **기본값은 RRF**로 둔다.
- **가중합** 결합은 동일 조건에서 **비교 실험·튜닝** 시 사용한다(기본 운영값은 RRF 유지).

### 6.4 chunk 검색 전략
- v1 구현에서는 하이브리드 검색을 지원한다.
- **기본값은 병렬 검색 후 결합**한다.
- **순차 검색**(예: 키워드 우선 후 벡터) 전략은 **실험·비교** 대상으로 구현·측정한다.

## 7. 문서 수집 및 인덱싱 명세

### 7.1 파일 등록
입력
- 업로드 파일
- 문서 메타데이터

처리 규칙
1. 원본 파일은 파일 스토리지에 저장한다.
2. 문서 메타데이터는 DB에 등록한다.
3. 등록 시 상태 기본값은 `pending(반영대기)`로 둔다.
4. 문서별 고유 식별자 `document_id`를 생성한다.

출력
- document_id
- 초기 상태값

### 7.2 포맷 정규화
- 최종 적재 포맷은 PDF로 통일한다.
- HTML/XLSX는 PDF로 변환 가능해야 한다.
- PDF는 필요 시 정규화한다.

### 7.3 PDF 파싱 및 OCR
필수 추출 정보
- 본문 텍스트
- 표
- 제목/소제목
- 문단 구조
- 페이지 정보

구현 원칙
- 단일 파서보다 본문/표/구조 분석을 분리한 다중 파서 전략을 우선한다.
- 스캔 PDF 또는 이미지 기반 PDF는 OCR을 수행할 수 있어야 한다.
- OCR은 검색 가능한 텍스트 레이어를 추가하는 방식으로 처리한다.

### 7.4 chunk 분할
원칙
- 의미 기반 분할을 기본으로 한다.
- 제목은 반드시 본문과 함께 묶는다.
- 표는 쪼개지지 않는 하나의 의미 단위로 유지한다.
- `page_start`, `page_end`, `heading_path`를 저장한다.

v1 chunking 우선순위
1. 조항 / 항 / 호
2. 제목 + 본문 문단 묶음
3. 표 전체 + 표 설명 문장
4. FAQ의 질문 1개 + 답변 1개

권장 길이 기준
- 기본 목표: 300~800 tokens
- 긴 절차 설명: 최대 1,000~1,200 tokens
- 아주 짧은 문단: 인접 문맥과 병합

[확인 필요] chunking 단위와 길이 기준은 방향은 정리됐지만, 일부 문서에서 여전히 `테스트 후 결정`으로 남아 있다. 운영 기준값으로 잠글지 회의에서 확정 필요.

### 7.5 임베딩 및 저장
- 각 chunk에 대해 임베딩 벡터를 생성한다.
- 임베딩은 `document_chunks`와 연결 가능해야 한다.
- 검색용 저장 완료 후 문서 상태를 갱신한다.

상태값
- pending
- processing
- completed
- failed
- reindex_required

반영 정책
- PDF 수정 반영은 전체 재색인 기본
- XLSX 변경분 부분 업데이트는 초기 범위에서는 운영 정책으로 별도 관리

## 8. 질문 처리 명세

### 8.1 라우팅
입력
- 사용자 자연어 질문
- 사용자 식별값

처리 규칙
1. 라우팅 전에 최소 핵심 정보를 먼저 추출한다.
2. 질문은 문서형 / 정형 조회형 / 혼합 조회형 / 처리 불가 중 하나로 분류한다.
3. 규칙 기반 분류를 우선하고, 필요 시 LLM 보조 분류를 사용한다.
4. 라우팅 실패 시, 미시도 route에 대해 최대 1회 재시도할 수 있다.
5. 모든 route 재시도 후에도 처리 경로를 결정하지 못하면 fallback으로 전환한다.
6. 질문 1건은 최종적으로 하나의 주 처리 유형으로만 분류한다.

### 8.2 핵심 정보 추출 및 정규화
추출된 정보는 문서 검색 필터 생성, 질의 재작성, DB/API 조회 파라미터 생성에 사용한다.

추출 대상
- 사람 이름
- 부서명
- 규정명/문서명
- 시점 표현(최신/올해/현재)
- 조건값(금액, 일수, 거리 등)

정규화 예시
- 법카 → 법인카드
- 반차 → 반일연차
- 여비 → 출장비

출력
- 정규화된 질의어
- 검색 필터 후보
- DB/API 파라미터

### 8.3 문서 검색(RAG)

#### Pre-Retrieval
- 질문 정규화/질의 재작성
- 하위 쿼리 분해
- 키워드 검색 질의 생성
- 의미 기반 검색 질의 생성
- 메타데이터 필터 생성

#### Retrieval
- 하이브리드 검색
- 키워드 검색: PostgreSQL Full Text Search
- 키워드 랭킹: 랭킹은 FTS 기본 랭킹으로 시작하고, BM25 계열 고도화는 후속 검토
- 유사도 검색: pgvector
- 메타데이터 필터 적용
- 문서 후보 검색 → chunk 후보 검색 → 결과 결합

#### Post-Retrieval
- 후보군 중복 제거
- 1차 점수화
- rerank
- 최종 context 편집

### 8.4 정형 데이터 조회
기본 전략
- v1에서는 사전 정의 SQL 또는 API 중심

지원 범위
- 이름 검색
- 유사 이름 검색
- 부서 검색
- 조직 구조 조회
- 연락처/이메일 조회
- 담당자/담당 부서 조회

보안 제한
- 조회 실패 시 부분 답변 또는 fallback
- 무제한 Text-to-SQL 미허용

### 8.5 혼합 조회
처리 규칙
1. 질문을 문서 조회 파트와 정형 데이터 파트로 분해한다.
2. 두 파트는 병렬 또는 순차로 실행할 수 있다.
3. 기본은 병렬 실행이다.
4. 의존 관계가 명확한 경우에만 순차 실행한다.
5. 문서형 결과는 근거 정보로 사용한다.
6. DB/API 결과는 현재 상태 정보로 사용한다.
7. 응답은 두 결과를 구분해서 제시한다.
8. 한쪽만 확보된 경우 부분 답변과 부족한 정보 안내를 함께 제공한다.
9. 둘 다 실패하면 fallback 응답을 제공한다.
10. 문서 기준일/시행일과 정형 데이터 조회 시점을 가능하면 함께 표시한다.

### 8.6 답변 생성
문서형 답변 포함 요소
- 문서명
- 조항/제목
- 페이지
- 최신 문서 여부 또는 기준일
- 관련 링크

정형 조회 답변 원칙
- 권한 범위 내 최소 정보만 포함
- 현재 상태 정보로 명시

혼합 조회 답변 원칙
- `기준/절차 안내`와 `현재 담당 정보`를 구분해서 제시
- 부분 답변 시 누락 축을 명시

### 8.7 fallback 및 예외 처리
fallback 발생 조건
- 검색 결과 부족
- 라우팅 실패
- 조회 실패
- 권한 없음

기본 메시지 원칙
- “정확한 답을 찾지 못했습니다” 수준의 안내형 메시지
- 가능하면 관련 문서 또는 관련 부서 탐색 경로 제공
- 허위 확정형 답변 금지

## 9. 데이터 구조 명세

### 9.1 documents
필수 컬럼 예시
- `document_id`: 문서를 구분하는 고유 ID
- `title`: 문서명
- `document_type`: 규정, 가이드, 매뉴얼 등 문서 유형
- `owner_department`: 문서 담당 부서
- `version`: 문서 버전 정보
- `effective_date`: 문서 시행일
- `base_date`: 문서 기준일
- `is_latest`: 최신 문서 여부
- `source_file_path`: 원본 파일 저장 위치
- `source_file_type`: PDF, HTML, XLSX 등 원본 파일 형식
- `status`: 문서 반영 상태
- `created_at`: 문서 등록 시각
- `updated_at`: 문서 정보 최종 수정 시각
- `error_code`: 반영 실패 시 오류 코드
- `error_message`: 반영 실패 시 오류 메시지
- `failure_reason`: 실패 원인 

### 9.2 document_chunks
필수 컬럼 예시
- `chunk_id`: chunk를 구분하는 고유 ID
- `document_id`: 원본 문서와 연결되는 ID
- `chunk_order`: 문서 내 chunk 순서
- `heading`: chunk 제목 또는 조항명
- `heading_path`: 상위 제목 경로
- `article_ref`: 조항/항/호 참조값
- `page_start`: chunk 시작 페이지
- `page_end`: chunk 끝 페이지
- `content`: 실제 검색 대상 본문
- `token_count`: chunk 길이 관리용 토큰 수
- `embedding`: 유사도 검색용 임베딩 벡터
- `created_at`: chunk 생성 시각
- `updated_at`: chunk 수정 시각


### 9.3 query_logs
저장 항목
- `request_id`: 요청 1건을 구분하는 고유 ID
- `user_id`: 요청 사용자 식별값
- `question`: 사용자 질문 원문
- `route_type`: 어떤 처리 경로로 분기됐는지 나타내는 값
- `status`: 처리 결과 상태
- `latency_ms`: 요청 처리 시간(ms)
- `error_code`: 오류 코드
- `error_message`: 오류 메시지
- `failure_reason`: 실패 원인
- `created_at`: 요청 발생 시각

### 9.4 retrieval_eval_logs
저장 항목
- `request_id`: 평가 대상 요청 ID
- `normalized_query`: 정규화된 질의
- `keyword_query`: 키워드 검색용 질의
- `semantic_query`: 유사도 검색용 질의
- `metadata_filter`: 적용한 메타데이터 필터
- `document_candidates`: 후보 문서 목록
- `chunk_candidates`: 후보 chunk 목록
- `final_context_chunk_ids`: 최종 context에 사용된 chunk ID 목록
- `rerank_applied`: rerank 적용 여부
- `created_at`: 평가 로그 생성 시각

### 9.5 공통 enum 정의

#### route_type
질문이 어떤 처리 경로로 분기되었는지 나타내는 값

- `rag`: 문서 검색(RAG) 중심 처리
- `db_api`: 정형 데이터(SQL/API) 중심 처리
- `hybrid`: 문서 검색 + 정형 조회 혼합 처리
- `unsupported`: 처리 불가 / 범위 외 질문

#### status
질문 처리 결과 상태를 나타내는 값

- `success`: 정상적으로 답변 생성 완료
- `fallback`: 충분한 근거가 없어 fallback 응답으로 전환
- `error`: 시스템 또는 처리 오류로 정상 응답 실패

#### document_status
문서 반영(ingestion) 처리 상태를 나타내는 값

- `pending`: 업로드 완료, 반영 대기 상태
- `processing`: 반영 처리 진행 중
- `completed`: 반영 완료
- `failed`: 반영 실패
- `reindex_required`: 재색인 필요 상태

#### failure_reason
실패의 상위 원인을 의미 기준으로 분류하는 값  
사용처: 사용자 질문 응답, 운영 로그, 문서 반영 실패 원인 기록

- `routing_failed`: 질문 라우팅 결정 실패
- `no_result`: 검색/조회 결과 부족
- `permission_denied`: 권한 부족으로 처리 불가
- `parse_failed`: 문서 파싱 실패
- `ocr_failed`: OCR 처리 실패
- `embedding_failed`: 임베딩 생성 실패
- `unknown`: 원인 분류 불가

#### error_code (시스템 처리/추적용 코드)
시스템 처리, 운영 추적, 디버깅을 위한 구체 오류 코드  
사용처: 운영 로그, 문서 상태 조회 API, 장애 분석

- `PERMISSION_DENIED`: 권한 부족
- `ROUTING_FAILED`: 라우팅 처리 실패
- `NO_RESULT`: 검색/조회 결과 없음
- `PARSE_FAILED`: 문서 파싱 실패
- `OCR_FAILED`: OCR 처리 실패
- `EMBEDDING_FAILED`: 임베딩 생성 실패
- `INDEX_SAVE_FAILED`: 인덱스 저장 실패
- `DB_QUERY_FAILED`: DB/API 조회 실패
- `INTERNAL_ERROR`: 기타 내부 시스템 오류

## 11. 품질 기준

### 11.1 성공 기준
- Recall@k를 우선 품질 지표로 사용
- 목표 Recall 최소 0.8 지향
- 문서형 답변은 근거 정보 포함
- 혼합 조회형 답변은 문서 근거와 현재 상태를 구분
- 관리자 API 또는 운영 도구로 업로드와 반영 상태 확인 가능
- 검색 결과 부족 시 fallback 응답 제공

### 11.2 비교 항목
- BM25/FTS 단독 vs 벡터 단독 vs 하이브리드
- RRF 적용 전후 및 **가중합 결합**과의 비교
- **병렬 검색 vs 순차 검색** 전략
- rerank 적용 전후
- 메타데이터 필터 적용 전후

### 11.3 평가셋 방향
- 자주 들어오는 질문 100~300개 수집
- 정답 문서 또는 정답 chunk 라벨링
- topN, topK, context 수 등 설정 비교

## 12. 구현 시 우선 결정해야 할 항목

**확정됨**
1. 운영 기능: **관리자만**. 관리자 업무는 **문서 업로드**와 **반영·처리 상태 확인**(실패 사유 포함)으로 한정.
2. 결과 결합: **RRF 기본**, **가중합**은 실험·비교용.
3. chunk 검색: **병렬 기본**, **순차**는 실험·비교용.
4. 동의어/사전: **운영 관리 기능은 v1 제외**. 코드 내 고정 치환만.

**미확정**
- chunking 기준을 운영 규칙으로 잠글지 여부
