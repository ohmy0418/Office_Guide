# PDF 문서 기반 RAG 시스템 기능 요건

## 문서 처리 및 로딩

- 대상 문서는 `doc.pdf` 파일 하나로 한다.
- **PDF 파일 경로 자동 탐지**: 개발 환경과 실제 환경 모두에서 동작하도록 여러 경로를 시도하여 PDF 파일을 찾는다.
  - 우선순위 순서:
    1. `services/doc.pdf` (현재 파일과 같은 디렉토리)
    2. `./services/doc.pdf` (작업 디렉토리 기준)
    3. `./src/services/doc.pdf` (작업 디렉토리 기준)
    4. `src/services/doc.pdf` (개발 환경)
    5. 기타 상대 경로들
  - 파일을 찾지 못한 경우 로그에 시도한 모든 경로를 출력하고, 빈 저장소로 초기화한다.
- 앱이 구동될 때 doc.pdf를 읽어 전체 내용을 메모리 DB에 다시 로드하고, 이전 벡터 정보는 초기화한다.
- PDF는 PyMuPDF(fitz)를 사용하여 페이지 단위로 텍스트를 추출한다.
- **전체 문서를 하나로 통합하여 청킹**: 페이지별로 추출한 텍스트를 전체 문서로 통합한 후, 의미 있는 문장/문단 단위로 청크를 생성한다. 이는 페이지 경계를 넘어서는 오버랩을 가능하게 하여 문맥을 보존한다.
- 각 청크에는 page_number(주요 페이지 번호), chunk_index(전체 문서 기준 인덱스) 메타데이터를 함께 저장한다.
- 각 청크에는 고유한 chunk*id를 부여한다 (형식: "chunk*{chunk_index}", 예: "chunk_0", "chunk_1").

## 텍스트 청킹

### 청킹 방식

- **전체 문서 통합 청킹**: 페이지별로 추출한 텍스트를 하나의 전체 문서로 통합한 후, 의미 있는 문장/문단 단위로 청크를 생성한다. 이 방식은 페이지 경계를 넘어서는 오버랩을 자연스럽게 허용하여 문맥을 보존한다.
- 추출된 텍스트는 **의미 있는 문장/문단 단위**로 우선 분할하며, 문맥을 최대한 보존한다.
- 청크 크기는 **문자 단위로 3KB(3072자)**를 기준으로 하되, **문장 경계를 우선적으로 존중**한다.
- 청크를 3KB로 split 할 때, 문맥 유지를 위해 **200자 overlap을 적용**하며, **페이지 경계를 넘어서도 오버랩이 적용**된다.
- LangChain의 RecursiveCharacterTextSplitter를 사용하여 텍스트를 분할하며, separator 우선순위를 통해 문장 단위 분할을 보장한다.
- **페이지 경계 제약 완화**: 페이지별 청킹이 아닌 전체 문서 통합 청킹을 통해, 청크가 페이지 경계를 자연스럽게 넘어갈 수 있다. 각 청크는 주요 페이지 번호를 메타데이터로 포함한다.

### 청킹 전략 상세

#### 의미 기반 분할 우선순위

1. **문단 단위** (`\n\n`): 빈 줄로 구분된 문단을 최우선으로 유지
2. **문장 단위** (`. `, `! `, `? `, `。`, `！`, `？`): 문장 부호로 구분된 문장 단위 유지
3. **줄 단위** (`\n`): 줄바꿈 단위 분할
4. **단어 단위** (` `): 공백 단위 분할
5. **문자 단위** (fallback): 위 방법으로 분할 불가 시에만 문자 단위 분할

#### 페이지 경계 처리

- **전체 문서 통합 청킹**: 페이지별로 추출한 텍스트를 하나의 전체 문서로 통합한 후 청킹을 수행한다. 이로 인해 청크는 자연스럽게 페이지 경계를 넘어설 수 있다.
- 청크는 **주요 페이지 번호**를 메타데이터로 포함한다 (청크의 중간 지점이 속한 페이지 또는 청크의 대부분이 속한 페이지)
- 페이지 경계에서 문맥이 끊기지 않도록, **오버랩이 페이지를 넘어갈 수 있다**
- 예: 페이지 10의 끝 부분과 페이지 11의 시작 부분이 하나의 청크에 포함될 수 있음
- 이 경우 청크의 `page_number`는 청크의 중간 지점이 속한 페이지 번호로 설정된다
- `chunk_index`는 전체 문서 기준 0-based 인덱스이며, 페이지를 넘어서는 연속된 번호를 가진다

### 텍스트 정리 및 전처리

PDF에서 추출한 텍스트는 청킹 전후에 정리 과정을 거친다:

- **불필요한 유니코드 문자 제거**: PDF 인코딩 문제로 발생하는 의미 불명의 유니코드 문자를 제거한다.

  - **허용 문자만 명시적으로 지정**: 정규식을 사용하여 허용 문자를 명시적으로 정의한다.
    - 영문자: `a-zA-Z`
    - 숫자: `0-9`
    - 한글: `가-힣` (완성형 한글), `ㄱ-ㅎ` (자음), `ㅏ-ㅣ` (모음)
    - 기본 구두점: `.,!?;:()[]{}"'\-` (공백 포함)
  - **제외되는 문자**: 위 허용 문자 목록에 없는 모든 유니코드 문자는 공백으로 치환된다.
    - 예: 한자 "堺", 기타 특수 유니코드 문자 등
  - **구현 방식**: `\w` (word character)를 사용하지 않고 명시적으로 허용 문자를 지정하여, 한자 등 불필요한 유니코드 문자가 포함되지 않도록 한다.
  - **청킹 시점 적용**: 청킹 전에 텍스트를 정리하고, 청킹 후 생성된 각 청크 텍스트에도 다시 정리 과정을 적용하여 이중 안전장치를 마련한다.
  - **출처 정보 표시 시 적용**: 사용자에게 출처 정보를 표시할 때도 정리된 텍스트를 사용하여 불필요한 문자가 표시되지 않도록 한다.

- **숫자 패턴 제거**: 페이지 번호나 메타데이터로 추정되는 반복 숫자 패턴을 제거한다.

  - 4자리 이상의 단독 숫자 제거 (예: "9999", "1234")
  - 3자리 이상의 단독 숫자 제거 (예: "999", "123")
  - 의미 있는 숫자는 보존 (예: "2024년", "3개", "12월")

- **공백 정리**: 여러 공백을 하나로 통합하고 앞뒤 공백을 제거한다.

정리 규칙은 `_clean_text()` 함수에서 구현되며, 다음 시점에 모두 적용된다:

1. PDF 로드 시: 페이지별 텍스트 추출 직후
2. 청킹 전: 전체 문서 통합 직전
3. 청킹 후: 각 청크 텍스트 생성 직후
4. 출처 정보 생성 시: 사용자에게 표시할 출처 텍스트 생성 시

## 한국어 텍스트 처리

- 토큰화는 공백 기반 split 후 조사 제거 방식을 사용한다 (별도 형태소 분석기 미사용).
- 다음 조사 목록을 토큰 끝에서 제거한다: `["은", "는", "이", "가", "을", "를", "에", "의", "로", "와", "과", "도", "만", "으로", "에서", "까지", "부터", "라고", "라는"]`
- 조사 제거는 검색 쿼리와 청크 텍스트 양쪽 모두에 동일하게 적용한다.
- "앞 3글자 기준" 어간 매칭은 완성된 한글 글자(음절) 기준으로 처리한다 (자모 분리 없음).

## 유사도 검색 시스템

- 다층 유사도 검색 알고리즘을 구현한다:
  - 완전 일치 (가중치 1.0): 정확한 키워드 매칭 (대소문자 구분 및 무시 모두 지원)
  - 부분 일치 (가중치 0.7): 어간 기반 매칭 (앞 3글자 기준, 대소문자 구분 및 무시 모두 지원)
  - 포함 관계 (가중치 0.5): 토큰 내 부분 문자열 매칭 (대소문자 구분 및 무시 모두 지원)
- **대소문자 무시 검색**: 영문 검색어의 경우 대소문자 구분 없이 매칭한다 (예: "IRIS Studio"와 "iris studio" 모두 매칭).
- 유사도 검색 시 threshold는 0.4 이상인 결과만 검색 후보로 포함한다.
- 사용자가 질문을 입력하면, 먼저 doc.pdf에서 생성된 벡터들을 대상으로 유사도 검색을 수행한다.

### 유사도 점수 계산 방식

- 쿼리의 각 토큰별로 청크 내 최고 매칭 점수를 구한다.
- 최종 청크 점수는 모든 쿼리 토큰의 매칭 점수 **평균**으로 계산한다.
- 예시: 쿼리 "인공지능 개발" → 토큰 ["인공지능", "개발"]
  - "인공지능" 완전일치 → 1.0
  - "개발" 부분일치("개발자"와) → 0.7
  - 최종 점수 = (1.0 + 0.7) / 2 = 0.85

### 검색 실패 조건 정의

- 다음 경우들을 "검색 실패"로 처리하여 일반 지식 기반 답변으로 전환한다:
  - 매칭되는 토큰이 전혀 없는 경우
  - 모든 청크의 점수가 threshold(0.4) 미만인 경우
  - 검색 결과의 최고 점수가 MIN_CONTEXT_SCORE(0.5) 미만인 경우
- 검색 결과가 있어도 최고 점수가 0.5 미만이면 일반 지식 기반 답변으로 전환한다.

### 검색 결과 품질 판단

검색 결과의 품질을 이중 임계값으로 판단한다:

- **SIMILARITY_THRESHOLD (0.4)**: 검색 후보로 포함되는 최소 점수
- **MIN_CONTEXT_SCORE (0.5)**: 컨텍스트로 사용하기 위한 최소 점수

검색 결과의 최고 점수가 MIN_CONTEXT_SCORE 이상일 때만 `has_context=True`로 설정하여 문서 기반 답변을 생성한다. 이는 낮은 관련성의 검색 결과로 인한 부정확한 답변을 방지한다.

## LangGraph 기반 워크플로우

### 워크플로우 개요

- RAG 플로우를 구현한다: `rag_search_node` → `generate_node` → `title_node` → `END`
- 각 노드는 실행 시점에 `loader()` 함수를 사용하여 사용자에게 상세한 진행 상태를 전달한다.
- 시스템 진입점은 `rag_search_node`이며, 노드 실행 시작 시점에 상태 메시지를 전송한다.
- 유사도 검색은 Top-3 결과를 가져오며, 각 결과에는 청크 텍스트와 page_number, chunk_index 정보가 포함된다.
- Top-3 검색 결과의 청크 텍스트들을 SystemMessage로 구성하여 LLM에게 컨텍스트로 전달한다.
- 답변 생성 완료 후 `title_node`에 메시지를 전달하여 대화 제목을 자동 생성한다.

### State 스키마 정의

LangGraph의 상태 관리를 위해 `AgentState` (기존 `MessagesState` 확장)를 사용한다:

```python
from typing import TypedDict, List, Optional, Dict, Any
from langchain_core.messages import BaseMessage

class ChunkResult(TypedDict):
    """검색된 청크 결과"""
    chunk_id: str           # 고유 청크 식별자, 형식: "chunk_{chunk_index}" (예: "chunk_0", "chunk_1")
    text: str               # 청크 텍스트 내용
    page_number: int        # 주요 페이지 번호 (1-based, 청크의 중간 지점이 속한 페이지)
    chunk_index: int        # 전체 문서 기준 청크 인덱스 (0-based, 페이지 경계를 넘을 수 있음)
    score: float            # 유사도 점수 (0.0 ~ 1.0)

class AgentState(MessagesState, total=False):
    """LangGraph 상태 (RAG 필드 포함)"""
    # 기존 필드들...
    search_results: Optional[List[Dict[str, Any]]]  # RAG 검색 결과
    has_context: Optional[bool]                      # RAG 검색 결과 존재 여부
    response_prefix: Optional[str]                   # RAG 답변 접두사
```

**중요**: `AgentState`에 RAG 관련 필드(`search_results`, `has_context`, `response_prefix`)가 정의되어 있어야 LangGraph가 상태를 올바르게 저장하고 전달할 수 있다.

### 사용자 피드백 및 상태 표시

#### loader 함수를 통한 상세 상태 정보 제공

사용자의 질의 이후 각 진행 단계에 대해 `loader()` 함수를 이용하여 상세한 상태 정보를 사용자에게 제공해야 한다.

**핵심 원칙:**
- 각 노드 실행 시점에 적절한 로딩 메시지를 제공하여 사용자가 현재 진행 상황을 명확히 인지할 수 있도록 한다
- 노드 내부에서 작업의 주요 단계별로 `loader()` 함수를 호출하여 단계별 상태를 전달한다
- 로딩 메시지는 구체적이고 의미 있는 내용으로 작성하여 사용자 경험을 향상시킨다

**구현 방식:**
- 별도의 `loading_node`를 생성하지 않고, 각 노드 함수 내에서 `loader()` 함수를 직접 호출한다
- `utils.loading` 모듈의 `loader()` 함수를 import하여 사용한다
- 노드 함수의 시그니처에 `config: RunnableConfig` 파라미터가 포함되어 있어야 한다

**각 노드별 상태 메시지 예시:**

| 노드              | 상태 메시지 예시                                    |
| ----------------- | --------------------------------------------------- |
| `rag_search_node` | "🔍 문서에서 관련 내용을 검색하고 있습니다..."       |
| `generate_node`   | "📝 검색 결과를 바탕으로 답변을 생성하고 있습니다..." |
| `title_node`      | "📌 대화 제목을 생성 중입니다..."                    |

**사용 예시:**

```python
from utils.loading import loader

async def rag_search_node(state: AgentState, config: RunnableConfig) -> dict:
    """
    사용자 쿼리에 대해 PDF 청크 유사도 검색을 수행한다.
    """
    # 검색 시작 시점에 상태 메시지 전송
    await loader("🔍 문서에서 관련 내용을 검색하고 있습니다...", config=config)
    
    # 실제 검색 로직 수행
    # ... 검색 로직 ...
    
    # 검색 완료 후 다음 단계로 진행
    return {"search_results": results, ...}
```

**주의사항:**
- `loader()` 함수는 `async` 함수이므로 반드시 `await` 키워드와 함께 사용해야 한다
- `config` 파라미터를 반드시 전달해야 한다 (없으면 로딩 메시지가 전송되지 않음)
- 하나의 노드 내에서 여러 번 `loader()` 호출이 가능하며, 각 단계별로 적절한 메시지를 제공할 수 있다
- 로딩 메시지는 다음 메시지가 오면 자동으로 대체되므로 별도의 종료 처리가 필요하지 않다

### 노드 정의

각 노드는 **단일 책임 원칙**을 따르며, 이름만으로 기능을 이해할 수 있어야 한다. **모든 노드는 실행 시점에 `loader()` 함수를 사용하여 사용자에게 상세한 상태 정보를 제공해야 한다.**

## 답변 생성 및 투명성

- 문서 기반 답변 시 "📄 문서 기반 답변:" 표시를 답변 앞에 포함하고, 접두사는 스트리밍으로 먼저 전송한다.
- 일반 지식 기반 답변 시 "🤖 일반 지식 기반 답변:" 표시를 답변 앞에 포함하고, 접두사는 스트리밍으로 먼저 전송한다.
- 유사도 검색 결과가 하나도 없으면, 사용자에게 "📄 문서에서 관련 내용을 찾지 못했습니다. 🤖 LLM 일반 지식을 바탕으로 답변을 생성하겠습니다..." 라는 사전 알림을 제공한다.

### 각주 및 출처 처리

- 각주는 **LLM이 직접 생성**하도록 프롬프트로 유도한다 (후처리 방식 아님).
- LLM에게 전달하는 컨텍스트에 `[1]`, `[2]`, `[3]` 번호를 미리 부여하여 제공한다.
- LLM이 답변 생성 시 해당 컨텍스트를 참조하면 본문에 각주 번호를 붙이도록 지시한다.
- **출처 정보는 시스템에서 별도로 생성**하여 제공한다 (LLM이 생성하지 않음).

#### 출처 정보 출력 로직

출처 정보는 다음 조건을 모두 만족할 때만 출력된다:

1. `has_context=True` (검색 결과의 최고 점수가 MIN_CONTEXT_SCORE 이상)
2. 검색 결과가 존재함 (`search_results`가 비어있지 않음)
3. LLM 응답에 정보 부재 표현이 없음 (예: "구체적인 내용이 없습니다", "정보가 없습니다" 등)

출처 정보 형식:

```
---
**출처**

[1] 페이지 {page_number}: {원문 내용 (최대 100자)}
[2] 페이지 {page_number}: {원문 내용 (최대 100자)}
[3] 페이지 {page_number}: {원문 내용 (최대 100자)}
```

출처 정보는 LLM 응답과 별도로 `PlainTextStream`을 사용하여 스트리밍으로 전송된다.

#### 정보 부재 감지

LLM 응답에서 다음 키워드가 감지되면 출처 정보를 출력하지 않는다:

- "포함되어 있지 않습니다", "정보가 없습니다", "구체적인 내용이 없습니다"
- "제공할 수 없습니다", "답변할 수 없습니다", "찾을 수 없습니다"
- "관련 내용이 없습니다" 등

이를 통해 검색 결과가 있어도 실제로는 질문에 대한 답변이 불가능한 경우 출처 정보가 출력되지 않도록 한다.

### 접두사 및 출처 정보 스트리밍 구현

- LLM 응답은 표준 `AIMessage`로 스트리밍 전송된다 (접두사 포함).
- 출처 정보는 `PlainTextStream`을 사용하여 별도 `AIMessage`로 추가 전송된다.
- 스트리밍 순서: 접두사 + LLM 답변 (각주 포함) → 출처 정보 (별도 메시지) → 완료
- 출처 정보는 LLM 응답과 분리되어 전송되므로 중복 표시되지 않는다.

## LLM 프롬프트 명세

### 컨텍스트 포맷

LLM에게 전달하는 SystemMessage 내 컨텍스트는 다음 형식을 따른다:

```

다음은 PDF 문서에서 검색된 관련 내용입니다. 이 정보를 바탕으로 답변해주세요.

[1] (페이지 {page_number})
{chunk_text}

[2] (페이지 {page_number})
{chunk_text}

[3] (페이지 {page_number})
{chunk_text}

```

### 답변 생성 지침

SystemMessage에 다음 지침을 포함한다:

```

- 반드시 한국어로 답변하세요.
- 제공된 컨텍스트 정보를 활용하여 답변할 때, 해당 부분에 [1], [2], [3] 형태의 각주 번호를 붙이세요.
- 컨텍스트에 없는 내용은 답변에 포함하지 마세요.
- 컨텍스트만으로 질문에 답변이 불가능한 경우, "제공된 정보에는 [질문 내용]에 대한 구체적인 내용이 포함되어 있지 않습니다"라고 명확히 알려주세요.
- 출처 정보는 시스템에서 별도로 제공되므로 답변에 "출처:" 섹션을 포함하지 마세요.

```

**중요**: LLM이 출처 정보를 생성하지 않도록 명시적으로 지시하며, 시스템에서 별도로 출처 정보를 추가한다.

### 일반 지식 답변 시 지침

검색 실패로 fallback할 경우, 다음 SystemMessage를 사용한다:

```

문서에서 관련 내용을 찾지 못했습니다. 일반 지식을 바탕으로 한국어로 답변해주세요.
단, 이 답변은 제공된 PDF 문서가 아닌 LLM의 학습 데이터에 기반함을 유의하세요.

```

## 상세 로깅 및 모니터링

- 각 질문 처리 시, 선택된 Top-3 결과의 유사도 점수와 page_number, chunk_index 정보를 콘솔 로그로 상세하게 남긴다.
- 벡터 검색 과정을 이모지와 함께 구조화된 로그로 기록한다:
  - 🔍 검색 시작: 쿼리, 토큰화 결과, 총 청크 수
  - ✅ 검색 완료: 결과 개수, 임계값 통과 개수, 처리된 청크 수
  - 📊 상위 검색 결과: 점수, 페이지 번호, 청크 ID, 텍스트 미리보기
  - 📈 검색 결과 품질 판단: 최고 점수, MIN_CONTEXT_SCORE, has_context 결정
  - 📝 출처 레퍼런스 생성: 생성된 출처 항목 개수
- `rag_search_node`와 `generate_node`에서 상태 전달 과정을 상세히 로깅한다.
- LLM 응답 분석 결과(정보 부재 감지 등)를 로깅한다.

## 설정 관리

- chunk 크기(3KB), overlap 크기(200자), 유사도 threshold(0.4), Top-K 값(3) 등은 `src/services/rag_config.py`에서 관리한다.
- 외부 YAML 파일(`config/rag_config.yaml`)은 생성하거나 참조하지 않으며, 설정 변경이 필요하면 `RAGConfig` 기본값을 수정한다.

### 설정 항목 전체 목록

```python
# rag_config.py에서 관리할 설정값

# 청킹 설정
CHUNK_SIZE = 3072  # 3KB 청크 크기 (문자 단위)
CHUNK_OVERLAP = 200  # 청크 간 오버랩 크기 (문자 단위)

# 검색 설정
SIMILARITY_THRESHOLD = 0.4  # 유사도 임계값 (이상인 결과만 선택)
TOP_K = 3  # 상위 K개 검색 결과 반환
MIN_CONTEXT_SCORE = 0.5  # 컨텍스트로 사용하기 위한 최소 점수 (최고 점수가 이 값 이상일 때만 has_context=True)

# 한국어 조사 목록 (토큰화 시 제거)
KOREAN_PARTICLES = [
    "은", "는", "이", "가", "을", "를", "에", "의", "와", "과",
    "으로", "로", "에서", "부터", "까지", "에게", "한테", "께",
    "처럼", "만큼", "같이", "도", "만", "조차", "마저", "까지",
    "부터", "에게서", "한테서", "께서", "으로서", "로서",
]

# PDF 파일 경로 (자동 탐지)
# 개발 환경과 실제 환경 모두에서 동작하도록 여러 경로를 시도하여 존재하는 파일을 찾음
# 우선순위: services/doc.pdf → ./services/doc.pdf → ./src/services/doc.pdf → src/services/doc.pdf 등
# 모듈 로드 시 자동으로 설정되며, 파일을 찾지 못한 경우 로그에 시도한 모든 경로를 출력
PDF_FILE_PATH = "{자동 탐지된 경로}"

# LLM 호출 설정
LLM_TIMEOUT_SECONDS = 30  # LLM API 호출 타임아웃 (초)
LLM_MAX_RETRIES = 3  # LLM 호출 최대 재시도 횟수
LLM_RETRY_BACKOFF_BASE = 2.0  # 재시도 백오프 지수 (exponential backoff)
```

### LLM 호출 안정성

- **타임아웃**: LLM API 호출 시 30초 타임아웃을 적용한다 (Constitution Principle 2 준수).
- **재시도 로직**: 실패 시 최대 3회까지 재시도하며, exponential backoff를 적용한다 (Constitution Principle 4 준수).
- 재시도 간격: 1초 → 2초 → 4초 (2.0의 지수로 증가).

## 아키텍처 및 구현

- 벡터 임베딩 처리와 DB 초기화 로직은 graph.py의 초기화 코드에서 실행되도록 하여, 앱 구동 시 자동으로 동작하게 한다.
- Constitution 준수를 위해 모든 RAG 관련 로직을 src/services/graph.py 내부에 구현한다.
- 메모리 기반 Vector DB를 사용하며, 현재는 키워드 기반 유사도 검색을 구현한다.
- 향후 확장을 위해 OpenAI 임베딩 또는 sentence-transformers와 FAISS 벡터 데이터베이스 도입을 고려한다.

## 개발 범위 제약

- `src/services/graph.py`는 초기 예제 구조를 참고하되, spec 요구사항에 맞춰 재구현한다.
- `.env`와 `src/core/config.py` 외부의 설정 파일은 수정하지 않으며, 추가 환경 변수는 `src/services/` 하위에서 관리한다.
